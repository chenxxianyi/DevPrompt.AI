# DevPrompt AI 下一阶段优化计划

更新时间：2026-05-10

## 当前基线

仓库当前已经具备可运行的三端结构：

- `web`：模板库、四类生成、历史记录、个人工作台、会员页
- `admin`：内容与配置管理后台
- `server`：认证、生成、统计、后台 API

同时已经具备：

- Docker Compose 启动 MySQL / Redis / API / Web
- Nginx 代理 `/api`
- 后端自动建库、迁移和种子数据初始化

## 优先级总览

| 优先级 | 事项 | 目标 |
| --- | --- | --- |
| P0 | 文案与编码一致性治理 | 清理仓库里残留的乱码文案和历史说明 |
| P0 | 用户端与后台功能边界补文档 | 防止文档超前于实现 |
| P1 | 试用申请后台页面 | 补齐 `/api/admin/trial-requests` 的前端管理入口 |
| P1 | 会员申请表单增强 | 在会员页补联系人和备注 |
| P1 | 模板元数据增强 | 提升模板可筛选性和复用效率 |
| P2 | Admin 独立部署方案 | 将后台纳入标准部署链路 |
| P2 | 测试与示例数据补充 | 降低联调和回归成本 |

## P0

### 1. 文案与编码一致性

现状：

- 仓库内部分中文内容存在历史编码问题
- README 与规划类文档容易和代码实际状态脱节

动作：

- 统一文档口径，以当前代码为准
- 后续逐步清理前端页面、Go 日志和种子文案中的乱码字符串
- 明确哪些功能是“已有 API，未做 UI”

完成标准：

- 根 README、模块 README、规划文档互相不冲突
- 新成员按文档可完成本地启动

### 2. 运行方式说明收口

现状：

- `docker-compose.yml` 不包含 `admin`
- `database/init.sql` 与运行时 `AutoMigrate + Seed` 并存

动作：

- 文档中统一说明部署入口与限制
- 强调当前以运行时建表和种子逻辑为准

完成标准：

- 不再把 `init.sql` 写成唯一初始化方式
- 不再误导用户认为 Compose 会自动部署后台

## P1

### 1. 试用申请后台页面

现状：

- 后端已有试用申请相关接口
- 管理后台缺少入口，无法直接运营跟进

建议实现：

- 新增后台菜单 `trial-requests`
- 支持分页、状态筛选、状态更新

验收标准：

- 运营可在后台查看申请列表
- 可将状态从 `pending` 更新为 `contacted` / `approved` / `rejected`

### 2. 会员申请表单增强

现状：

- 用户端会员页当前只传 `planCode`
- 已预留 `contact`、`message` 字段，但前端没有输入控件

建议实现：

- Pro / Team / Enterprise 方案增加联系人和备注输入
- 提交后显示更明确的反馈

验收标准：

- 后台可看到用户补充信息
- 用户端不再只有“无上下文申请”

### 3. 模板元数据增强

现状：

- 模板已经支持分类、详情、收藏和点赞
- 但适用工具、场景、示例输入输出仍不足

建议实现：

- 增加模板适用工具字段
- 增加示例输入输出
- 后台支持编辑这些元数据

验收标准：

- 用户能更快判断模板用途
- 模板到生成器的带入链路更顺畅

## P2

### 1. 管理后台标准化部署

建议：

- 为 `admin` 增加 Dockerfile 和部署入口
- 明确后台是独立域名还是 `/admin` 子路径

### 2. 测试与演示数据

建议：

- 为关键接口补充最基本的集成测试
- 提供更稳定的开发演示数据方案

## 推荐迭代顺序

1. 文档与文案一致性
2. 试用申请后台页面
3. 会员申请表单增强
4. 模板元数据增强
5. Admin 标准部署
